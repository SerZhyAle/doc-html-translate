# Research: EPUB → HTML + перевод (Windows CLI)

## Project Links
- GitHub Pages: https://serzhyale.github.io/doc-html-translate/
- GitHub Repository: https://github.com/SerZhyAle/doc-html-translate

## 1) Контекст задачи

Требуется консольная утилита (без GUI), которая:
- регистрируется в Windows как обработчик файлов `.epub`;
- по открытию `.epub` извлекает содержимое в каталог рядом с файлом;
- формирует HTML-структуру, отражающую внутреннюю структуру EPUB (включая изображения, шрифты, стили);
- опционально переводит текстовое содержимое через Google Translate;
- открывает результат в браузере;
- поддерживает CLI-флаги (`-no-translate`, `-src`, `-dst`, регистрация).

ОС: Windows. Язык реализации: Go.

---

## 2) Способы решения (варианты по подсистемам)

## 2.1 Регистрация обработчика `.epub` в Windows

### Вариант A: HKCU (текущий пользователь)
- Механика: запись в `HKCU\\Software\\Classes`.
- Плюсы: не нужны права администратора, проще UX.
- Минусы: действует только для текущего пользователя.
- Вывод: базовый и предпочтительный путь для персонального инструмента.

### Вариант B: HKLM (вся система)
- Механика: `HKLM\\Software\\Classes`.
- Плюсы: глобальная регистрация.
- Минусы: UAC/админ-права, выше риск конфликтов с системными политиками.
- Вывод: не обязателен для текущей задачи, может быть расширением.

---

## 2.2 Извлечение и разбор EPUB

EPUB — это ZIP-архив с обязательным `META-INF/container.xml`, указывающим на `content.opf`.

### Вариант A: Ручной разбор (ZIP + XML стандартной библиотекой)
- Инструменты: `archive/zip`, `encoding/xml`.
- Плюсы: минимальные зависимости, полный контроль.
- Минусы: больше кода и edge-cases на стороне проекта.
- Риски: ошибки в обработке нестандартных EPUB.

### Вариант B: Использование готовой библиотеки EPUB
- Плюсы: быстрее старт, меньше низкоуровневого кода.
- Минусы: качество/поддержка библиотек в Go неравномерны, возможны ограничения форматов.
- Риски: vendor lock-in, сложность отладки чужой логики.

### Рекомендация для ресерча
- Для production-стабильности сравнить 2–3 библиотеки на наборе реальных EPUB.
- Если ни одна не покрывает требования по spine/nav/assets — переходить на ручной разбор.

---

## 2.3 Построение HTML-результата

### Вариант A: Сохранение структуры «как в EPUB» + индексная страница
- Сущность: копировать XHTML/ресурсы и строить `index.html` как точку входа по `spine`.
- Плюсы: минимум трансформаций, ниже риск сломать вёрстку.
- Минусы: неоднородное качество рендера между книгами.

### Вариант B: Полная нормализация в единый HTML
- Плюсы: единый формат выхода.
- Минусы: высокий риск поломок CSS/ссылок/якорей, сложная трансформация.

### Вывод
- Для первой версии практичнее Вариант A (минимально инвазивный подход).

---

## 2.4 Перевод через Google Translate

### Вариант A: Cloud Translation API v2 (REST)
- Плюсы: простой REST-интерфейс, предсказуемая интеграция.
- Минусы: лимиты, стоимость, качество на сложной технике/формулах.

### Вариант B: Cloud Translation API v3
- Плюсы: более богатые сценарии (глоссарии и др.).
- Минусы: выше интеграционная сложность.

### Способ обработки HTML при переводе
1. Парсить HTML в DOM.
2. Извлекать только текстовые узлы (исключая `script/style`).
3. Отправлять сегменты на перевод пакетами.
4. Возвращать переведённый текст в исходные узлы.
5. Сохранять HTML без изменения структуры/атрибутов.

### Ключевой риск
- Превышение лимитов запросов и/или размера payload.
- Нужны чанкинг, ретраи, backoff и ограничение параллелизма.

---

## 2.5 Запуск результата в браузере

### Вариант A: Браузер по умолчанию (ShellExecute)
- Плюсы: корректно для пользовательской среды, без жёсткой зависимости от Chrome.
- Минусы: различия поведения браузеров.

### Вариант B: Жёстко Chrome
- Плюсы: предсказуемость рендера.
- Минусы: Chrome может быть не установлен/в другом пути.

### Вывод
- Практический default: браузер по умолчанию ОС.

## 3) Основные проблемы и риски

## 3.1 Технические
- EPUB2/EPUB3 различаются по навигации и метаданным; нужна толерантность к «грязным» книгам.
- В книгах встречаются относительные пути, невалидный XHTML, нестандартные CSS/шрифты.
- Перевод HTML легко ломает документ, если затронуть теги/атрибуты вместо текстовых узлов.
- Большие книги создают долгую обработку и высокое потребление памяти.

## 3.2 Операционные
- Ошибки регистрации ассоциации файлов (права, политика ОС, конкурирующие программы).
- Отсутствие сети/прокси/ограничения firewall при обращении к Google API.
- Квоты и стоимость API при массовой обработке.

## 3.3 Безопасность
- Хранение ключа API в бинаре — риск компрометации (даже для личного инструмента).
- Работа с недоверенными EPUB: риск path traversal при распаковке ZIP.
- Возможные вредоносные HTML-фрагменты в исходной книге.

## 3.4 UX/надежность
- Пользователь ожидает прогресс; «тишина» в консоли выглядит как зависание.
- На ошибках нужна понятная диагностика: шаг, причина, действие пользователя.

## 4) Нефункциональные требования (для фиксации)
