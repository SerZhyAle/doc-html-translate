## Git-режим разработки (обязательный)

- Репозиторий: локальный Git, одиночная ветка `master` (без feature-веток).
- Публикация репозитория во внешние remote не планируется.
- Цель Git: полный локальный трекинг прогресса и возможность отката к любому коммиту.
- Коммит-политика: после каждого успешного шага, как только шаг собирается (`scripts/build.ps1`) — делать коммит.
- Формат шага: `код/конфиг -> успешная сборка -> коммит -> запись в журналы`.
- Документация проекта хранится в каталоге `DEV/`.
- Устаревшая/закрытая документация переносится в `temp/docs-archive/`.
- Журналы:
    - `dev/CHANGELOG.md` — факт изменения файлов (операционный журнал).
    - `DEV/COMMIT_LOG.md` — hash коммита + сообщение + список файлов в коммите.
- Автоматизация: `scripts/commit_after_build.ps1 -Message "..."`.

## Принятые решения (от пользователя)

### Git и безопасность
- R1: Используется только внутренний локальный `git`-репозиторий.
- R2: Репозиторий не публикуется, приоритет — скорость разработки и история откатов.
- R3: Для v1 допустимо хранение чувствительных проектных данных в репозитории (в рамках локального использования).
- R15: Коммиты нужны только для возможности роллбека, не ритуальные.
- R18: Никаких ограничений безопасности. Программа для личного пользования.

### Поведение программы
- R4 (Q1): Если выходной каталог уже существует → сразу открыть готовый HTML, не перегенерировать.
- R5 (Q2): При ошибке перевода → поставить прогресс на паузу, показать сообщение в shell-окне.
- R9 (Q6): Автозакрытие консоли при успехе. При ошибке — окно остаётся открытым.
- R10 (Q7): Best-effort: пропуск проблемных частей EPUB с предупреждением.
- R11 (Q8): Максимум ~100 МБ. Картинки — норма, извлекаем и кладём.

### Конфигурация и API
- R6 (Q3): API-ключ Google — hardcoded в коде.
- R7 (Q4): Только HKCU (текущий пользователь).
- R8 (Q5): Браузер по умолчанию Windows.
- R13 (Q10): Дефолт: src=en, dst=ru. Другие языки — через флаги.

### Контент и перевод
- R12 (Q9): Структура книги сохраняется «как в оригинале» (заголовки, страницы). Шрифты заменяются на доступные если оригинал отсутствует.
- R16 (Q13): Первая рабочая версия — без перевода (только конвертация + открытие в браузере).

### Логирование и тесты
- R14 (Q11): Пользователь предоставит тестовые EPUB в `test_epub/`.
- R17 (Q14): Логирование — только для дебага. В релизе — никакого внешнего логирования.

## Фаза 5: Поддержка PDF (текстовых)

**Цель**: конвертация текстовых PDF в HTML с тем же UX что и EPUB (навбар, зум, перевод, браузер).

**Статус**: В РАБОТЕ ◄

### Архитектурные решения (PDF)

- R19: PDF → HTML использует ту же цепочку что и EPUB начиная с шага 2 (`index.html` + navbar + translate + browser).
- R20: Библиотека: `github.com/ledongthuc/pdf` (fork rsc/pdf, BSD-3, pure Go, поддержка text extraction по строкам).
- R21: Каждая страница PDF → отдельный `page_NNN.html` в выходном каталоге.
- R22: PDF-экстракция возвращает `*epub.Book` (adapter): Title = имя файла, Manifest/Spine = сгенерированные страницы. Это позволяет использовать `htmlgen`, `navbar`, `translateContent` без изменений.
- R23: Определение формата (`epub` / `pdf`) — по расширению файла в `pipeline.go`.
- R24: OCR для сканированных PDF — TODO на будущее. Пока только текстовые PDF.

### Deliverables
- `doc-html-translate.exe file.pdf` → создаёт `<file>/index.html` + `page_NNN.html` → navbar + translate → browser.
- Пустые страницы PDF пропускаются (не генерируют HTML).
- `-no-translate`, `-force`, `-src`, `-dst` работают одинаково для обоих форматов.
- Регистрация `.pdf` как обработчика — опционально (через `-register`).
- Unit-тесты для PDF text extraction.
